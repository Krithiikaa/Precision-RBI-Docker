§ 01

The core idea

An ordinary browser does something dangerous a thousand times a day: it downloads code written by strangers — JavaScript, fonts, trackers, exploits — and runs it on your machine. Remote browser isolation breaks that habit. The stranger's code runs somewhere else, on a server we control and throw away, and the only thing that crosses back to the user is a picture of the result.

The requirement that shaped this build was strict: the user keeps their normal browser and normal habits. They type irctc.com into the same address bar as always. Behind that one action, the site is fetched, executed, and rendered on our Precision-RBI server, and streamed back as a live image — with no manual steps and no plugin to open.

✦

The real reason

You cannot be harmed by code you never execute. Isolation does not try to detect whether a site is malicious — an arms race nobody wins. It assumes every site is hostile and moves the execution off the user's device entirely. Safety by architecture, not by guessing.

§ 03

The five services

Proxy
mitmproxy · 8080
The single door. Intercepts each request, checks if the device is signed in, logs the URL, swaps every page navigation for a streaming bootstrap.
Gateway
FastAPI · 8000
Brain of the system. One-time login, leases isolated browsers, serves the viewer, relays the pixel stream so the host stays private.
Browser Host
Playwright · 9000
Where danger is contained. Disposable Chromium fetches the real site, emits JPEG frames, accepts forwarded clicks and keystrokes.
Redis
session store
Remembers which device belongs to which user, and which isolated browser is leased to which session.
ClickHouse
audit log
Append-only record of every URL every user reached, with timestamps — the compliance backbone.
The seam
principle, not a box
The line the page's code can never cross. Everything else exists to keep that line intact.

§ 05

What happens at each level

Three layers make the whole thing run: the terminal that starts the services, the operating system that quietly redirects all traffic and trusts our certificate, and the browser that the user actually touches. Configure the OS once and every browser inherits it — that is the secret behind "any browser, no fiddling."

§ 06

The certificate principle

To read an HTTPS request and render it on the server, the proxy must decrypt it. TLS is designed to make that impossible for a middleman — which is exactly what the proxy is. The only legitimate way through is for the user's browser to trust the proxy's own certificate authority.

So Precision-RBI runs its own private certificate authority. On a managed device that CA is installed into the trust store — silently, by policy. From then on the proxy can present a valid certificate for any site, the browser accepts it, and the interception is invisible.

⚠

The weight of this key

That certificate authority can impersonate any website to every device that trusts it. It is the most sensitive secret in the system — which is why isolation belongs only on devices an organisation owns and administers, never imposed on people who didn't consent. The power that makes it work is the same power that demands it be guarded.

Why big sites needed an extra step

Large sites ship a flag inside the browser saying "only ever trust real certificates for me" (HSTS preload). When the proxy intercepted them, the browser raised MITM_DETECTED and refused. The fix — enterprise-root trust — tells the browser that interception by its own administrator is legitimate. On managed devices this is a single pushed policy.

§ 07

How pixels travel back

The server browser captures itself with Chrome's screencast feature, emitting a JPEG frame each time the page changes. Frames travel browser host → gateway relay → the user's canvas. Input travels the reverse path as small coordinate and key events.

This is the honest source of any sluggishness: whole JPEG frames, base64-encoded, crossing two hops. The production answer is WebRTC video encoding, which sends only what changed.

§ 08

What is built — and what comes next

Everything in this briefing works today: transparent interception from an unmodified browser, one-time login, full URL capture, disposable isolated rendering, a live pixel stream with working input, and handling for both ordinary and HSTS-protected sites. It is a genuine, end-to-end remote browser isolation system.

Three things separate this MVP from a production product, in order of impact:

  1. WebRTC streaming — replaces JPEG-over-two-hops with hardware video encoding. The single largest improvement to how it feels to use.
  2. Session & pool management — reuse one isolated browser per user instead of leasing per tab, with idle eviction and autoscaling; retire the legacy direct-portal door.
  3. Edge features — downloads scanned before release, clipboard control, and an HSM-backed certificate authority so the trust anchor is never a loose file.
